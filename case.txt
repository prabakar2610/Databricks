### Troubleshooting Azure Databricks Private Link DNS and Connectivity Issues

Here are focused troubleshooting steps and commands to help diagnose problems with DNS resolution and private connectivity in your Azure Databricks Private Link setup, especially when the domain resolves to a public IP instead of a private one. These cover private DNS configuration, zone linkage, Databricks NCC and endpoint setup, and load balancer diagnostics.

---

#### 1. DNS Resolution Checks

- Run NSLookup or Dig from different points in your network to verify which DNS path resolves the domain:

```bash
nslookup <your-domain>
dig <your-domain>
```
- Run the above commands from:
  - The Azure Databricks notebook (using Python subprocess or Databricks utilities)
  - An Azure VM inside the target private VNet
  - A host outside Azure (to validate public DNS vs. private DNS result)

- Compare the returned IP addresses. The resolution from Databricks and the VM should yield the private IP of your load balancer frontend if configured correctly.

---

#### 2. Private DNS Zone Linkage Verification

- In Azure Portal, navigate to your Private DNS Zone and check:
  - The record set for `<your-domain>` points to the intended private IP.
  - The private DNS zone is linked to the correct virtual networks (check under "Virtual network links").
  - The link is set as "enabled" for auto-registration if applicable.

- Use Azure CLI to check your DNS records and virtual network links:
```bash
az network private-dns zone show --name <your-zone> --resource-group <your-rg>
az network private-dns record-set a list --zone-name <your-zone> --resource-group <your-rg>
az network private-dns link vnet list --zone-name <your-zone> --resource-group <your-rg>
```

---

#### 3. Load Balancer Diagnostics

- Confirm health probe status:

```bash
az network lb probe show --resource-group <your-rg> --lb-name <your-lb> --name <your-probe>
```

- Confirm backend pool membership and rule configuration:

```bash
az network lb address-pool show --resource-group <your-rg> --lb-name <your-lb> --name <your-backend-pool>
az network lb rule list --resource-group <your-rg> --lb-name <your-lb>
```

- Validate that the frontend IP configuration is set to an internal (VNet) IP, **not public**.

---

#### 4. Databricks NCC and Private Link Endpoint State

- In Azure Databricks Account Console, ensure your NCC is tied to the correct workspace and region, and the private endpoint’s status shows **ESTABLISHED**.

- Re-review the allowed domain names in the endpoint rule match exactly what you expect to resolve (no wildcards and no DNS redirects/chasing).

---

#### 5. End-to-End Connectivity and Port Test

- From your Databricks notebook, run a Python connectivity test:

```python
import socket

def test_connectivity(host, port):
    try:
        with socket.create_connection((host, port), timeout=5):
            print("Connection successful")
    except Exception as e:
        print(f"Failed: {e}")

test_connectivity('<your-domain>', <port>)
```

- If this fails, trace which firewall or NSG block might exist by reviewing Network Security Groups (NSGs) for both the Databricks VNet and your backend VNet.

---

#### 6. NCC Azure Resource ID and Domain Name Auditing

- Double check the Azure Resource ID used in the NCC matches the Private Link service resource ID from Azure, as documented:

```
/subscriptions/<subscription-id>/resourceGroups/<resource-group-name>/providers/Microsoft.Network/privateLinkServices/<private-link-service-name>
```
- Ensure the domain names specified in the NCC are the **exact FQDNs** you want resolved via the private link.

---

#### 7. Review for DNS Chasing or Redirects

- Note that DNS chasing/redirect (CNAME chains resolving beyond the initial label) are not supported. Make sure you do not use CNAMEs that ultimately resolve to a public IP—your DNS record in the private zone should **directly** point to the private frontend IP.

---

### Next Steps

If public IPs continue to resolve:
- Check for missing or misconfigured VNet links to your private DNS zone.
- Remove public DNS records for the same FQDN to avoid fallback.
- Add an explicit A record for your domain in the private DNS zone.

Thank you for the detailed context. Based on your description, this is not a generic “misconfiguration” symptom — it points to a very specific failure mode in **Azure Databricks Serverless + Private Link to customer-managed internal services**. Below is a structured diagnosis and a concrete troubleshooting path.

---

## Executive summary

If **DNS resolution from Databricks Serverless resolves to public IP instead of private IP**, then **Private Link is not being used at all**, regardless of whether:

* NCC shows “Established”
* Private endpoint was approved
* Domains are configured
* Load balancer is healthy

This almost always means one of the following:

1. The **Private DNS Zone is not correctly linked to the Private Endpoint**, or
2. The **Private DNS Zone name does not match exactly what Databricks expects**, or
3. The **Private Endpoint NIC is not registered in the zone**, or
4. The **NCC domain configuration is incomplete or syntactically incorrect**, or
5. The customer is unintentionally testing from a workspace that is **not actually using Serverless with that NCC**.

---

## How the architecture actually works (important nuance)

In this pattern:

Databricks Serverless → Microsoft-managed VNet → Private Endpoint → Customer VNet → ILB → Backend VM

Databricks **does not resolve DNS using your VNet**.
Instead, **Databricks injects DNS overrides internally** based on:

* The **NCC domain configuration**
* The **private endpoint + private DNS zone**
* The **validated Private Link resource**

If DNS is resolving publicly inside the notebook, then Databricks has **not accepted the domain as eligible for private routing**.

---

## High-confidence root cause

From your symptoms:

> "resolution of that domain was public instead of private"
> "All private records resolved to same IP (likely Databricks NAT)"
> "No traffic ever hits the load balancer"

This is consistent with:

> **Databricks did not associate the domain with the private endpoint.**

In other words:
The NCC says “Established” for the private link resource, but the **domain binding is failing silently**.

This happens when:

### ❌ Common mistake #1: Wrong DNS zone name

The private DNS zone **must match exactly the FQDN suffix**.

Example:

* Service FQDN:
  `api.internal.contoso.com`
* Private DNS zone must be:
  `internal.contoso.com`
  (NOT `contoso.com`, NOT `privatelink.internal.contoso.com`)

If this does not match exactly, Databricks will:

* Accept the domain in NCC UI
* But **not inject private routing**
* And DNS will resolve publicly

---

## What to validate step-by-step

### 1. On the Private Endpoint NIC

Go to:

> Private Endpoint → Network Interface → IP configuration

You should see:

* A private IP
* And under DNS:

  * A reference to the private DNS zone

If you **do not see DNS zone associated here**, the private endpoint is not linked to DNS properly.

---

### 2. In the Private DNS Zone

Check records:

You must have either:

* A-record:

  ```
  api.internal.contoso.com → 10.x.x.x
  ```

OR

* Wildcard:

  ```
  *.internal.contoso.com → 10.x.x.x.x
  ```

If the record exists but resolution is still public from Databricks, then the failure is not DNS itself — it is **Databricks not honoring the domain**.

---

### 3. In NCC → Domains

This is extremely sensitive.

Correct:

```
internal.contoso.com
```

Wrong:

```
https://internal.contoso.com
api.internal.contoso.com
*.internal.contoso.com
```

Only the **zone**, no protocol, no wildcard, no host.

---

### 4. Confirm the workspace is actually using that NCC

This is often overlooked.

Check:
Workspace → Settings → Network connectivity config

If the workspace was created before NCC, or attached incorrectly, it will:

* Show private link established
* But **still egress publicly**

A fast test:
From notebook:

```python
import requests
print(requests.get("https://ifconfig.me").text)
```

If IP belongs to Azure public Databricks ranges → NCC is not applied.

---

## Why everything looked "healthy" but still failed

Azure side components can all be green:

* LB healthy
* Private endpoint approved
* DNS zone created
* VM reachable internally

But Databricks will **only route traffic privately if all three match**:

* Private Endpoint
* Private DNS Zone
* NCC domain suffix

Any mismatch → silent fallback to public routing.

---

## Specific test that should work when fixed

Once correctly configured:

From Databricks notebook:

```python
import socket
print(socket.gethostbyname("api.internal.contoso.com"))
```

Must return:



If DNS is private but TCP fails, only then does LB/backend become suspect.
In your case, DNS never became private → upstream config issue.
